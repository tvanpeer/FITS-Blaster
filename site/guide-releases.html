Release & Issue Workflow

A cheat sheet for managing releases, beta builds, and issue tracking with GitHub.

Release channels

FITS Blaster uses tag names to distinguish stable releases from beta builds. Everything happens on the main branch — no extra branches to manage.

TagExampleWhat happens
v1.23Stable releaseHomepage updated, appcast.xml updated, full deploy
v1.23-beta.1Beta buildGitHub pre-release, appcast-beta.xml updated, beta page updated

Releasing a beta

1
Make your changes and commit them to main
2
Bump the version in Xcode to e.g. 1.23-beta.1
3
Push with a tag from Xcode, or from the terminal:
git tag v1.23-beta.1 && git push origin v1.23-beta.1
4
The CI pipeline builds, signs, notarises, and publishes the DMG as a GitHub pre-release. The homepage stays on the last stable version.

Promoting to stable

1
When you're happy with the beta, bump the version to 1.23 (drop the -beta.N suffix)
2
Push with tag: git tag v1.23 && git push origin v1.23
3
The CI pipeline updates the homepage, changelog, and appcast.xml. All users get the update.

Issue tracking

Issues live at github.com/tvanpeer/FITS-Blaster/issues. Use them to track bugs, feature requests, and questions.

Labels

LabelUse for
bugSomething is broken
featureNew functionality
improvementEnhancement to existing feature
questionNeeds investigation or user question

Milestones

Group issues into a release by assigning them to a milestone (e.g. v1.23). When the release ships, close the milestone.

Common commands

# Create a bug report
gh issue create -t "Crash on 3-axis FITS cubes" -l bug -m v1.23

# Create a feature request
gh issue create -t "Histogram overlay" -l feature -m v1.23

# Create an improvement
gh issue create -t "Faster thumbnail loading" -l improvement -m v1.23

# List all open issues
gh issue list

# List issues for a specific milestone
gh issue list -m v1.23

# View a specific issue
gh issue view 5

# Close an issue (if not auto-closed by commit)
gh issue close 5

Auto-closing issues from commits

Include Fixes #5 (or Closes #5) in a commit message, and GitHub will automatically close issue #5 when the commit lands on main.

git commit -m "Handle 3-axis FITS cubes gracefully. Fixes #5"

Milestone lifecycle

# Create a new milestone for the next release
gh api repos/tvanpeer/FITS-Blaster/milestones \
  -f title="v1.24" -f state=open

# Close a milestone when the release ships
# (replace 1 with the milestone number)
gh api repos/tvanpeer/FITS-Blaster/milestones/1 \
  -f state=closed

# List all milestones
gh api repos/tvanpeer/FITS-Blaster/milestones

Putting it all together

1
Plan: Create issues for what you want in v1.23, assign them to the v1.23 milestone
2
Build: Work on the features. Use Fixes #N in commits to auto-close issues
3
Test: Push v1.23-beta.1 tag to get a test build. Fix bugs, push v1.23-beta.2, etc.
4
Ship: Push v1.23 tag for the stable release. Close the milestone.
5
Repeat: Create a v1.24 milestone and start the next cycle
